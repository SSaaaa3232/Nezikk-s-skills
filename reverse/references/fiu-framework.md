# FACTS / INFERENCES / UNKNOWNS 框架

## 定义

### FACTS（已交叉验证）

满足以下**至少两个条件**才可归入 FACTS：
1. 源码中直接可见（有文件路径 + 模块编号 + 函数名）
2. 动态调试确认（断点 / console / HAR 抓包验证）
3. 两个独立证据来源交叉验证

```
✅ 好例子:
"TRON 签名消息使用完整 SIWE 格式"
  证据1: 26057-xxx.js module 28386 (源码)
  证据2: HAR 抓包中 signature message 字段匹配

❌ 坏例子:
"后端使用 JWT 验证"
  → 只看到 Authorization: Bearer xxx，没看到源码证据，归入 INFERENCES
```

### INFERENCES（合理推断）

基于 FACTS 的合理推断，但**缺少直接证据**。

```
✅ 好例子:
"服务端可能不检查 isRecharged 标志来决定模型访问权限"
  基于: 错误处理代码中无 isRecharged 相关逻辑 (FACT)
  缺少: 未直接看到服务端中间件代码 (所以是推断)
```

### UNKNOWNS（未确认）

需要进一步调查才能确定的问题。

```
✅ 好例子:
"生产环境使用的是 PGLite 还是 PostgreSQL 模式？"
"服务端是否有速率限制或异常检测？"
"积分扣费是预扣还是后扣？"
```

## 使用原则

1. **不猜测 FACTS**：如果只有一个证据来源且未交叉验证，降为 INFERENCE
2. **UNKNOWNS 要可操作**：每项 UNKNOWN 都应该暗示一个具体的验证方法
3. **FACTS 要精确**：每个 FACT 必须附带证据锚点
4. **数量不限**：FACTS 可以很多，INFERENCES 可以很多，UNKNOWNS 也可以很多——重要的是分类准确

## 分类检查清单

在写报告前，问自己：

- [ ] 这个结论是「我看到代码这样写的」还是「我推测它应该是这样」？
- [ ] 如果 10 分钟后面试官问我「你怎么确定」，我能给出两个独立证据吗？
- [ ] 有没有一个简单的动态测试可以验证这个推断？（如果有 → 去做，把它变 FACT）
